## Patch boundaries

This train corrects the seven confirmed v1.5.0 findings. It does not begin v1.6.0 and does not add an adapter, third schema, new default write format, dependency, network behavior, provider pricing, or cost engine.

Implementation proceeds one commit-sized chunk at a time. After each chunk, stop at the maintainer gate. Version changes, a changeset, tag, publish, and GitHub release remain Gate D maintainer actions and are not part of the four implementation commits below unless separately authorized.

## Chunk 1 — Version and canonical metadata read path

### Confirmed problem

- The public CLI hard-codes and publishes `1.0.0` from a `1.5.0` package.
- `extractMetadata`, list, stats run-level aggregation/correlation, and search run-level filters are v0.1-only.
- Existing v0.2 command tests cover `what` and timeline, not list/stats/search parity.
- Returning an embedded v0.2 `runId` will break any code that reopens a file by assuming `<runId>.jsonl`; the current canonical fixtures already demonstrate the mismatch.

### Implementation approach

1. Inject the root public package version into the CLI build and register that value with Commander. Keep the private CLI workspace version out of user-visible output. Add both program-level and built-artifact assertions to prevent drift.
2. Make metadata extraction use the shared dual-format parser. Derive run ID, name, status, start/end, duration, and source event count consistently from normalized data while keeping file size and creation time from the filesystem.
3. Add a path-based canonical read helper so stats/search can reopen the exact file in `TraceMetadata.filePath`; do not assume the embedded run ID equals the basename.
4. Remove the separate v0.1-only stats correlation scan. Read correlation/group metadata from the same normalized events/path used for summary data.
5. Add v0.1/v0.2 parity fixtures and exercise list, stats, search, correlation, status, name, since/duration, event count, and filename/embedded-ID mismatch.

### Compatibility risks

### Security implications

Metadata extraction remains read-only and local. Do not log event contents on parse failures. Correlation filters compare normalized metadata without exposing values. No network behavior is added.

## Chunk 2 — Report safety and normalization correctness

### Confirmed problem

- `report --redaction-profile share|strict` redacts only the execution tree; what/timeline content is derived from unredacted events.
- Mixed v0.1/v0.2 files are reordered by format during normalization.
- Persisted `error.name` is mislabeled as a v0.1 stack.
- Existing tests do not cover these safety and fidelity cases.

### Implementation approach

1. Create profile-redacted, bounded copies of structured trace events before building any report section. Build title/what/timeline/tree/errors/attributes from the safe copy. Reuse the existing `Redactor` and profile definitions; do not use post-render regex replacement. Preserve input immutability.
2. Apply profile semantics consistently: share redacts its correlation/customer identifiers everywhere; strict additionally redacts prompt/output/message-like data everywhere. Keep local behavior compatible.
3. Convert each valid JSONL row inline in `parseTraceJsonl`, appending any one-to-many normalized rows at that source position. Retain format detection, warnings, and the persisted-row result.
4. Map a persisted error to v0.1 with its message and only a real string `attributes.errorStack` when present. Never use `error.name` or `error.code` as stack text.
5. Update report CLI help from “tree section” to “entire report.”

### Exact files likely to change

- `packages/core/src/report.ts`
- `packages/core/src/read-trace.ts`
- `packages/core/src/persisted/to-trace-event.ts`
- `packages/core/src/exporters/redact-export.ts` or a narrowly named sibling helper for structured trace-event copies
- `packages/cli/src/index.ts`
- `packages/cli/src/report.ts` only if option plumbing or output metadata changes

### Tests

- `packages/core/test/report.test.ts`: Markdown and HTML whole-report redaction, local parity, input immutability.
- `packages/cli/test/report.test.ts`: CLI share/strict option behavior and help wording.
- `packages/core/test/redaction-profiles.test.ts` and/or `security-redaction.test.ts`: correlation, request ID, prompt/output/message, error message, nested data, and no source mutation.
- `packages/core/test/read-trace.test.ts`: exact mixed ordering, invalid-line behavior, and adjacent one-to-many expansion.
- `packages/core/test/persisted/to-trace-event.test.ts`: error name is not stack; a real legacy `attributes.errorStack` round-trips.
- Keep HTML escaping, event-size, path traversal, export redaction, and v0.1 migration suites green.

### Fixtures

- Add `packages/core/test/fixtures/mixed-v0.1-v0.2-order.jsonl` with interleaved synthetic rows.
- Add a synthetic report-safety fixture only if it improves reuse; use obvious fake sentinel strings, never real credentials or customer data.
- Reuse `fixtures/traces-v0.2/manual-tool-error.jsonl` for native error-name coverage.

### Compatibility risks

- Share/strict report content intentionally changes and may replace identifiers or strict-profile messages with `[REDACTED]`.
- Mixed-file consumers that accidentally relied on format-grouped output will observe source-preserving order; source order is the correct contract.
- Persisted error names/codes are not representable in v0.1 `ErrorInfo`; they remain available in persisted data but are not fabricated into a compatibility field.

### Security implications

This is the security-critical chunk. Test the complete Markdown and HTML documents, not just the tree fragment. Confirm redaction precedes rendering, HTML escaping still occurs after redaction, source events are unchanged, no secrets enter fixtures, and no new disk/network behavior is introduced.

## Chunk 3 — Token vocabulary completion

### Confirmed problem

- v0.2-to-v0.1 inspection normalization drops a supplied token `total`.
- v0.1 `TokenMetadata` omits both `total` and `cached`; v0.2 omits `cached`.
- Converter behavior is inconsistent: one path preserves `total`, while other paths recompute or discard it.
- The accepted RFC and fixtures do not cover cached-token vocabulary.

### Implementation approach

1. Add optional non-negative `total` and `cached` to the v0.1 token metadata model and optional non-negative `cached` to `PersistedTokenUsage`.
2. Preserve valid supplied `input`, `output`, `total`, and `cached` in every converter. Derive `total = input + output` only when total is absent and both components are present. Do not add cached a second time.
3. Extend summaries/reports to aggregate and expose total/cached without changing existing input/output fields. Define behavior for partial data explicitly and keep output deterministic.
4. Do not introduce provider APIs, token counting, pricing tables, currency, billing semantics, or network calls.
5. Do not add a canonical cost field: the approved repository vocabulary has none. Leave arbitrary existing attributes opaque and do not promote them into usage totals.

### Exact files likely to change

- `packages/core/src/types.ts`
- `packages/core/src/types/persisted-inspect-event.ts`
- `packages/core/src/persisted/from-trace-event.ts`
- `packages/core/src/persisted/to-trace-event.ts`
- `packages/core/src/persisted/from-inspect-event.ts`
- `packages/core/src/persisted/to-inspect-event.ts` if explicit validation/normalization replaces the current spread
- `packages/core/src/trace-metadata.ts`
- `packages/core/src/what.ts`
- `packages/core/src/report.ts` only for display/typing integration
- `packages/core/src/index.ts` and persisted entry barrels only if type exports require adjustment

### Tests

- `packages/core/test/types.test.ts` and `types/persisted-inspect-event.test.ts`: optional fields and non-negative finite validation.
- All four converter test files under `packages/core/test/persisted/`: supplied total preservation, derived total fallback, cached preservation, partial usage, invalid values, round trips.
- `packages/core/test/read-trace.test.ts`: v0.2 normalization preserves all supported counts.
- `packages/core/test/trace-metadata.test.ts`, `what.test.ts`, and `report.test.ts`: multi-step aggregation and deterministic rendering.
- `packages/langchain/test/metadata.test.ts` and callback tests remain green and continue proving cost fields are ignored.

### Fixtures

- Extend `fixtures/traces/llm-with-tokens.jsonl` with supplied `total` and `cached`.
- Extend `fixtures/traces-v0.2/llm-tokens-and-streaming.jsonl` with `cached` while retaining its supplied `total`.
- Update fixture READMEs and validation expectations. Counts must be synthetic and internally coherent.

### Compatibility risks

- Additive optional properties can change deep-equality snapshots and JSON output when source data supplies them.
- Aggregation must distinguish “absent” from zero and avoid double-counting cached tokens.
- Validators accepting `cached` widen the valid v0.2 shape without changing the schema version; existing files remain valid.

### Security implications

Token counts are not secrets, but remain subject to event bounds when carried in metadata. Do not capture prompt/output text, raw chain-of-thought, provider credentials, or new full-response payloads. No provider lookup or network access is permitted.
