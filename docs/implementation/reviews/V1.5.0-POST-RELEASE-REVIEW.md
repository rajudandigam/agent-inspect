## Detailed evidence

### 1. Runtime CLI version

`packages/cli/src/index.ts` registers Commander with a literal `1.0.0`. Neither the CLI build configuration nor tests derive/assert the public root package version. The root public manifest is `1.5.0`; the private CLI workspace manifest remains `0.1.0`, so the public root version must be the intentional source.

Existing CLI tests check the program name, description, commands, and help, but do not assert `program.version()` or execute the built `--version` path.

### 2. v0.2 metadata behavior

`extractMetadata` reads each line and immediately skips it unless `isTraceEvent(parsed)` succeeds. `isTraceEvent` accepts only `schemaVersion: "0.1"`. `list`, `stats`, and `loadTraceMetadataList` for search all depend on this function. In addition, `stats` has a separate `readRunStartedMetadata` helper that also accepts only v0.1.

Against the canonical v0.2 fixture directory, the built CLI reported:

```text
list:   five rows, all status=unknown, eventCount=0, no v0.2 run names/times/durations
stats:  totalRuns=5, success=0, error=0, running=0, unknown=5, no run durations
search --status success --json: []
```

`stats` still counted ten normalized steps and `search --kind ...` can still find normalized steps. This is why the finding is not described as total v0.2 failure: the defect is the run-metadata and filtering path.

The fix must not make normalized `runId` assume the filename matches the embedded v0.2 `runId`. Current fixtures deliberately use names such as `manual-basic.jsonl` with embedded `run_manual_basic`. A path-based shared reader is needed for stats/search after metadata begins returning the embedded ID.

### 3. Whole-report redaction

`buildRunReport` currently performs these operations:

1. builds and renders `what` from the original events;
2. builds and renders the timeline from the original events;
3. converts events to a tree and applies `redactRunTreeForExport` only to that tree.

The CLI help and `docs/CLI.md` accurately expose the current unsafe limitation by calling the option “tree section” redaction, but that behavior is not sufficient for a report offered with `share` and `strict` profiles.

A synthetic strict-profile reproduction returned the same result for Markdown and HTML:

```json
{"correlationLeaked":true,"requestLeaked":true,"runErrorLeaked":true}
```

This is a security regression because `correlationId` and `requestId` are explicit `share` profile keys, and `message` is an explicit `strict` profile key. Redaction must happen on structured copies before any report section is derived; post-render string replacement would be brittle and unsafe.

### 4. Mixed-file ordering

For a file containing a v0.2 event named `persisted-first` followed by a v0.1 run named `legacy-second`, normalization produced:

```json
{"input":["p-first","legacy-second"],"output":["legacy-second","persisted-first"]}
```

The current test only asserts `format === "mixed"` and a minimum event count. It never asserts source ordering. Conversion should occur inline per valid input row; any one-to-many rows produced from a v0.2 event should remain adjacent at that row's source position.

### 5. Persisted error conversion

The v0.2 error model contains `name`, `message`, and `code`; the v0.1 `ErrorInfo` contains `message` and optional `stack`. The current converter fabricates the stack from the error class name:

```json
{"message":"boom","stack":"TypeError"}
```

The inverse v0.1-to-v0.2 converter already preserves a real legacy stack under `attributes.errorStack`. The corrective projection should restore only a valid `attributes.errorStack`; a v0.2 name or code has no lossless v0.1 field and must not be mislabeled.

### 6. Token vocabulary and conversion

The canonical v0.2 fixture supplies:

```json
{"input":1200,"output":356,"total":1556}
```

The inspection normalization returns only:

```json
{"input":1200,"output":356}
```

The accepted vocabulary document explicitly deferred cached tokens, but the corrective scope now requests `cached` where supplied. The patch should support optional, non-negative `input`, `output`, `total`, and `cached` counts across the two canonical representations and converters. A supplied valid `total` should be preserved; it may be derived from input plus output only when absent. Cached tokens are a breakdown and must not be added to total a second time.

No approved cost vocabulary exists in the repository. The RFC says cost/USD fields are never a core pricing feature, and LangChain tests require provider cost fields to be ignored. Therefore v1.5.1 must not introduce a cost field, pricing table, provider lookup, or calculation. Arbitrary pre-existing attributes remain opaque attributes; they are not promoted to canonical token usage or report billing semantics.

### 7. Stale documentation and release evidence

Confirmed stale or incomplete locations include:

- `docs/CLI.md`: says `what` is v0.1-only and v0.2 support “lands” in v1.5.0 Chunks 7–8; says `report` is v0.1-only; describes redaction as tree-only.
- `packages/cli/src/index.ts`: runtime help also describes report redaction as tree-only.
- `ROADMAP.md`: the Now section still centers the v1.5.0 train and the maintainer table marks it **In progress (planning)** despite the same file stating 1.5.0 was published.
- `docs/implementation/CURSOR-MAINTAINER-ROADMAP.md`: calls v1.5.0 the current planning train.
- `docs/implementation/V1.5.0-RELEASE-READINESS.md`: status says published, but every post-publish checkbox remains unchecked.
- `docs/LIMITATIONS.md`: contains mixed old and new v0.2 CLI descriptions and understates the affected metadata path as only `list`.

## Coverage assessment

The repository has good v0.1 unit coverage and targeted v0.2 coverage for `what` and `timeline`, but the release gate missed these cases:

- no built CLI or Commander version assertion;
- no v0.2 cases for `extractMetadata`, list, stats, status search, time/name filters, or correlation filters;
- no report test with `share`/`strict` across what, timeline, errors, title/tree, Markdown, and HTML;
- no mixed-format ordering assertion;
- no assertion that error name is distinct from stack;
- no v0.2-to-v0.1 preservation assertion for `total`, and no `cached` vocabulary tests;
- no post-publish checklist completion gate tied to evidence.

## Compatibility, security, and product-boundary assessment

- **Schema:** v0.1 and v0.2 remain readable. Optional token fields are additive; no third persisted format and no write-path switch are planned.
- **API:** existing root and subpath exports remain. Any new token properties are optional.
- **CLI:** output becomes more correct for v0.2. JSON consumers may observe real run IDs/names/status/times instead of filename fallbacks and `unknown`; this is intended but should be called out.
- **Security:** whole-report redaction is the highest-priority correction. Tests must cover both report formats and prove the input events are not mutated.
- **Dependencies/package size:** no new dependency is needed. Version injection should use build-time data already in the repository.
- **Network:** no runtime network behavior, telemetry, provider token lookup, or pricing lookup is permitted.
- **Write behavior:** manual trace writing remains `schemaVersion: "0.1"`; no destructive migration is permitted.
